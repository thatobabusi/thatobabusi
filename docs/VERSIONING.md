# Versioning

This repo treats the profile like a product and versions it with
[Semantic Versioning](https://semver.org/): **`vMAJOR.MINOR.PATCH`**.

Since a profile has no API, the numbers are mapped to how visible and
structural a change is rather than to breaking changes:

| Bump | When | Examples |
|:-----|:-----|:---------|
| **MAJOR** (`v2.0.0` → `v3.0.0`) | Full visual redesign or structural overhaul of the README | New banner/theme, section layout rebuilt, stats stack replaced |
| **MINOR** (`v2.0.0` → `v2.1.0`) | A new section, feature, or automation is added | New workflow, new featured project, docs folder added, automated changelog |
| **PATCH** (`v2.1.0` → `v2.1.1`) | Fixes and small content tweaks | Typos, broken image swaps, badge color changes, reading-list updates |

Day-to-day content edits (books finished, article list refreshed) do **not**
require a release — they land on `main` and show up in the
[auto-generated changelog](CHANGELOG.md). Cut a release when the profile has
meaningfully changed since the last tag.

## Where the version shows up

One tag drives everything:

1. **Git tag** — annotated `vX.Y.Z` tag on `main`
2. **GitHub Release** — created from the tag with curated notes
3. **Release badge** — the purple `release` badge at the top of the profile
   reads the latest GitHub Release automatically
4. **[RELEASES.md](RELEASES.md)** — hand-written history of what each version
   contained
5. **[CHANGELOG.md](CHANGELOG.md)** — auto-generated commit-by-commit record
   (regenerated by the Changelog workflow on every push; never edited by hand)

The split to remember: **RELEASES.md is curated, CHANGELOG.md is generated.**
Release notes say *why* a version exists; the changelog records *everything
that happened*.

## Cutting a release

1. Land all changes on `main` and push.
2. Add a section for the new version at the **top** of the history in
   [RELEASES.md](RELEASES.md) (newest first), commit, push.
3. Tag and publish:

   ```bash
   git tag -a v2.2.0 -m "Profile v2.2.0 - short summary"
   git push origin v2.2.0
   gh release create v2.2.0 --title "Profile v2.2.0" --notes "One-paragraph summary with highlights"
   ```

4. Verify the `release` badge on the profile shows the new version
   (shields.io caches for a few minutes).
